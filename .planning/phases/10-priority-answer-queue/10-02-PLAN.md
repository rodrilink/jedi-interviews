<objective>
Close the single failed Phase 10 must-have (D-11 / truth 8): a finished request's late gateway
event must never bleed into the next queued entry. The two-lane priority queue runs requests
back-to-back — when request 1 reaches a terminal, `clearActive()` + `startNext()` synchronously make
request 2 active within the same handler invocation — so a straggler `text` (or a duplicate
`done`/`error`) from request 1's shared-emitter stream, arriving AFTER request 2 is active, currently
passes the `this.active === undefined` guard (request 2 IS active) and corrupts request 2's
buffer/history/push under request 2's id. This is committed review finding CR-01 (critical) + WR-01
(warning), and it is the exact late-delta bleed the phase's threat model (T-10-05) exists to prevent.

The fix threads the reserved `requestId` through the gateway so each of the four handlers
(`text`/`done`/`error`/`abort`) can positively match the event's originating request to
`this.active.requestId` and drop it otherwise — mirroring the `flushDelta` and `startCodeChallenge`
requestId guards already in the same file. The prior plan (10-01) deferred this as "out of scope"
because the interface did not carry a requestId (see 10-01-SUMMARY.md §Deviations); the verifier ruled
it IN scope for the D-11 truth, so this gap-closure plan performs the minimal interface change that
makes the positive match possible.

Purpose: make the D-11 no-cross-bleed invariant real (not just for the idle gap) before Phase 11
wires a real, higher-frequency auto-trigger onto the same queue — where back-to-back execution and
straggler races become routine, not incidental.
Output: a threaded requestId on the gateway seam, four positively-guarded handlers, and a
two-request-in-sequence regression test that fails on the current code and passes after the fix.
</objective>

<decision_design>
## Chosen approach (Claude's Discretion, per gap instructions): tag the event with its requestId (approach a)

The gateway is a shared `EventEmitter` subscribed ONCE (`handlersWired`), and its events carry NO id.
A naive closure over "the latest requestId" cannot work for the shared once-wired handlers, and a
per-stream re-subscription (approach b) would force rewriting every `gateway.emit(...)` in the suite
into per-stream `stream.emit(...)` AND making `IAiStream` an event source — a larger structural and
interface change. Approach (a) is the cleaner, more minimal seam here:

1. Add `requestId: number` to `IAiPromptRequest`. The orchestrator ALREADY holds the reserved
   `requestId` at both `gateway.stream(...)` call sites (`startRequest` :376/:384, `startCodeChallenge`
   :404/:434), so threading it is a one-field addition — no new state.
2. Add a trailing `requestId: number` parameter to the four `IAiGateway.on(...)` listener signatures.
   `AnthropicGateway.stream()` closes over `request.requestId` and passes it on every `emit(...)`.
3. Each of the four orchestrator handlers gains the positive guard
   `eventRequestId !== this.active.requestId` (in addition to the existing `active === undefined`
   check), identical in spirit to `flushDelta`/`startCodeChallenge`. A req1-tagged straggler arriving
   while req2 is active now fails the positive match and is dropped.

This keeps the interface change minimal (one field + a trailing param on four events), keeps the
shared-emitter subscription model unchanged (no structural rewrite), and confines the behavior change
to the four handlers. The existing 29 tests are updated mechanically to pass the active request's
requestId on their emits (read from `gateway.stream.mock.calls[n][0].requestId`) so they stay green.
</decision_design>
