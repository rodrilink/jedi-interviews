<objective>
Refactor `AiOrchestrator` from the current single-in-flight "drop-or-cancel-if-busy" model into a **priority answer queue** so a manual answer and a (future, Phase 11) auto-answer can both be requested without racing. This is the enabling backend layer for the rest of milestone v1.2.

Purpose: Reverse the v1.1 cancel-on-re-press semantics (D-01/D-02/D-03) — nothing cancels an in-flight stream; new requests enqueue and run in order. Manual requests take priority over queued autos (two-lane FIFO, D-05). A mode-keyed debounce collapses accidental bursts (D-06) and a single-in-flight gate guarantees at most one `stream()` call at a time (D-07). A bounded cap with drop-oldest-auto eviction keeps the queue from growing unbounded in a long meeting (D-08/D-09). Delivered as a **pure main-side refactor with NO auto-trigger source yet** — the auto lane is implemented so Phase 11 can feed it, but Phase 10 only exercises it via tests.

Output: A refactored `ai-orchestrator.ts` (queue + run loop + debounce + cap) and an extended `ai-orchestrator.test.ts` (new queue tests, reversed cancel-semantics tests, retained green regression tests). No renderer/UI change, no `index.ts` wiring change, no new files, no new dependency.

This satisfies AA-05 (priority queue replacing drop-if-busy; manual preempts autos; nothing cancels in-flight) and AA-06 (debounce + single-in-flight so a burst never spawns parallel Claude calls).
</objective>
