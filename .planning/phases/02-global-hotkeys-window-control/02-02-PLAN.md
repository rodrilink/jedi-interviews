<objective>
Wire the real keyboard control loop: build a `WindowControlActionsService` whose handlers mutate the overlay window — show via the existing `showOverlay()` wrapper / hide via a new `hideOverlay()`, move ~50px per step clamped at the outermost virtual-desktop edge (crossing monitors allowed), step opacity in 10% increments within a 20%->100% range, toggle the HUD content flag, and quit via `app.quit()` — then inject those handlers into the `HotkeyRegistrarService` from 02-01 and extend `IOverlayStatus` with the main-owned `hudVisible` flag so the renderer reflects HUD visibility as a pure view.

This is the second vertical slice: after this plan, the user can press hotkeys while a meeting app holds focus and SEE the overlay show/hide, move, fade, toggle its HUD, and quit — a complete, demonstrable keyboard control loop (OVL-03 opacity, OVL-05 show/hide, CTL-01 move). The chords remain the 02-01 placeholders; 02-03 finalizes them after conflict testing. Hold-to-repeat for move/opacity is already wired in 02-01's registrar (per uiohook key-repeat events); the handlers in this plan are per-step and idempotent so repeated invocation simply steps again.

Purpose: Turn the registered chords into observable window mutations — the working control loop.
Output: `WindowControlActionsService` + tests, the hide path + `hudVisible` + `isOverlayVisible` in `overlay-window.manager.ts`, handlers injected into the registrar, and HUD visibility driven by the pushed flag.
</objective>
