<objective>
Stand up the global hotkey layer: install and rebuild the first native module in the repo (`uiohook-napi`), build a `HotkeyRegistrarService` that registers the locked five-group Ctrl+Alt action set as a passive low-level hook (with a `globalShortcut` fallback behind a single seam), check each registration result at startup, and surface the aggregated outcome as a persistent `Hotkeys:` line in the existing DebugHud over the read-only `jedi:status` channel.

This is the first vertical slice of the keyboard control loop: after this plan, pressing the show/hide chord while a meeting app holds focus produces an observable effect in main (a stub handler logs/marks the action) and the HUD truthfully reports how many chords registered — proving CTL-02 (hotkeys fire while another app is focused) and CTL-03 (failures surfaced, never silently dropped). The concrete window mutations (move, opacity, real show/hide, quit) are wired in 02-02; this plan establishes the registrar, the result-aggregation shape, the IPC status extension, and the HUD readout.

Purpose: Establish the hotkey registration + failure-surfacing seam that 02-02's window-control actions plug into.
Output: `uiohook-napi` installed + rebuilt against Electron 35.7.5, `HotkeyRegistrarService` + tests, `IOverlayStatus.hotkeys`, and the HUD `Hotkeys:` status line.
</objective>

|----------|-------------|
| OS keyboard -> uiohook-napi (main) | The passive low-level hook observes app-wide keystrokes; only specific Ctrl+Alt chords are acted on. |
| main -> renderer (jedi:status) | One-way, read-only, non-secret status push. No renderer->main control channel exists or is added. |
| npm install (uiohook-napi, @electron/rebuild) | New native + dev dependencies enter the supply chain; native code is compiled and loaded into main. |
